## 2026-07-06 — ADR-018: отложенная отправка — поллинг в процессе API, без Celery/Redis/cron

**Решение:** доставка запланированных сообщений (`messages.scheduled_at`) реализована
как бесконечный `asyncio`-цикл (`app/services/scheduler.py::run_scheduler_loop`),
запускаемый через FastAPI `lifespan` в том же процессе, что и сам API — раз в 15 секунд
сканирует `messages WHERE scheduled_at <= now() AND scheduled_broadcast_at IS NULL` и
рассылает найденные. Не Celery/RQ/внешний cron.

**Контекст:** в проекте уже принято (ADR-010, `docs/02-ARCHITECTURE.md` §4.2), что при
одном инстансе API Redis не обязателен — real-time и так работает через in-process
`ConnectionManager`. Заводить отдельный воркер (Celery+Redis/RQ) только ради одной фичи
отложенной отправки — непропорционально объёму задачи и добавляет вторую движущуюся
часть (ещё один процесс, который надо деплоить и мониторить) там, где хватает цикла с
`asyncio.sleep`. Точность доставки — до 15 секунд, что для «отправить сообщение позже»
(в отличие от, например, звонка) не является ощутимой проблемой.

**Последствия:**
- **Не масштабируется горизонтально без доработки.** Если API когда-нибудь будет
  работать в нескольких инстансах (см. условие про Redis в ADR-010), каждый инстанс
  будет крутить свой цикл и может продублировать доставку одного и того же сообщения
  (гонка между `SELECT ... WHERE scheduled_broadcast_at IS NULL` и `UPDATE` в разных
  процессах). На один инстанс (текущий демо-деплой на Render) это не проблема. При
  масштабировании — либо `SELECT ... FOR UPDATE SKIP LOCKED`, либо перенос на
  Celery beat/RQ scheduler, либо на управляющий инстанс.
- Тестируется без реального ожидания: `deliver_due_scheduled_messages(db)` — отдельная
  функция «один проход», принимает сессию и не трогает время самостоятельно; тесты
  создают сообщение с `scheduled_at` в будущем через API, затем руками переводят
  `scheduled_at` в прошлое через ту же тестовую сессию и вызывают проход напрямую —
  не нужен реальный `sleep()` в CI (`backend/tests/test_scheduled_messages.py`).
- `run_scheduler_loop` использует свой отдельный `create_async_engine(..., poolclass=NullPool)`,
  а не общий `app.db.session.engine` — тот же приём, что и в `tests/conftest.py`
  (ADR-010): фоновая задача не должна делить пул соединений с обработчиками запросов.

## 2026-07-06 — ADR-017: Lighthouse PWA ≥ 90 не проверен из этой среды — нет Chrome/headless-браузера

**Решение:** не заявляю фактический Lighthouse-скор — вместо этого зафиксирован чек-лист
того, что уже сделано для соответствия критериям PWA-инсталлируемости, и явно указано,
что нужен реальный прогон Lighthouse (Chrome DevTools → Lighthouse, на задеплоенном
HTTPS-URL) после демо-деплоя.

**Контекст:** в этой среде нет `chrome`/`chromium` бинарника (`which chrome` пусто),
только `npx lighthouse` (CLI ставится, но самого браузера для аудита нет) — попытка
реального прогона ничего не даст, а придумывать цифру было бы враньём.

Сделано (проверяемо по коду, не по факту прогона):
- `manifest.webmanifest`: `name`, `short_name`, `icons` (192/512 + maskable),
  `start_url`, `display: standalone`, `theme_color`/`background_color` — все поля,
  которые Lighthouse проверяет для критерия «installable» (ADR-015).
- Service worker регистрируется и активно перехватывает запросы (`injectManifest`,
  precache app shell) — критерий «has a registered service worker».
- `apple-touch-icon`, `theme-color` meta — iOS-специфичные, не входят в сам Lighthouse
  PWA-скор, но нужны для «Add to Home Screen» на iPhone (ТЗ §1).
- HTTPS — обеспечивается платформой (Vercel даёт HTTPS по умолчанию), не кодом.

**Последствия:** после демо-деплоя на Vercel нужно руками прогнать Lighthouse
(Chrome DevTools → Lighthouse → PWA) на реальном HTTPS-адресе и завести отдельную
задачу, если скор ниже 90 — конкретные причины (например, размер бандла, LCP) можно
будет увидеть только там, спекулировать бессмысленно.

## 2026-07-06 — ADR-016: Render остаётся (backend+WS), нельзя свести демо-деплой к «только Vercel + Supabase»

**Решение:** заказчик попросил убрать также и Render, оставив исключительно
Vercel + Supabase. После явного уточнения — оставляем Render как есть; убирается только
Neon (см. ADR-013), Render как хостинг FastAPI+WebSocket остаётся обязательным.

**Контекст:** бэкенд держит одно постоянное WS-соединение на клиента через
in-process `ConnectionManager` (`backend/app/ws/`) — real-time сообщения, typing,
presence, read-receipts. Serverless/Edge-функции Vercel (включая Python-рантайм)
принципиально не держат соединение между запросами: функция завершается сразу после
ответа. Перенести весь бэкенд на Vercel без Render означало бы либо полностью убрать
WebSocket и переписать real-time на поллинг/Supabase Realtime (большая переработка
`backend/app/ws/*` и `frontend/src/shared/ws/*`, вне текущего объёма работ), либо
взять другой хостинг с долгоживущим процессом вместо Render (Fly.io/Railway/VPS) —
пользователь явно выбрал вариант «оставить Render, но расписать инструкцию
подробнее», не меняя архитектуру.

**Последствия:** `docs/03-SETUP-DEPLOY.md` §3 дополнен явным разделом «Почему не
только Vercel + Supabase» с этим же обоснованием, чтобы не возникал этот вопрос заново
при следующей ревизии деплоя. `render.yaml` и `backend/Dockerfile` не привязаны к
конкретно Render — при желании сменить хостинг на другой с тем же свойством
(долгоживущий процесс), `Dockerfile` переносится без правок, `render.yaml` просто не
используется. Заодно `.env.example`/`docs/03-SETUP-DEPLOY.md` обновлены под реальный
SMTP-провайдер — `smtp.bz` с доменом-отправителем `spi-2015.ru` (пользователь сообщил,
что уже настроил это на стороне smtp.bz/DNS) — см. §1.1 в `docs/03-SETUP-DEPLOY.md`.

## 2026-07-06 — ADR-015: service worker — vite-plugin-pwa с `injectManifest`, не `generateSW`

**Решение:** PWA собирается через `vite-plugin-pwa` (`frontend/vite.config.ts`) в режиме
`injectManifest` с собственным service worker'ом `frontend/src/sw.ts`, а не декларативным
`generateSW`. Причина — нужен `workbox-background-sync` (очередь исходящих сообщений при
отсутствии сети из ТЗ §2.6), а это требует ручного кода воркера, а не только конфигурации
`runtimeCaching`. `sw.ts` также сразу содержит обработчики `push`/`notificationclick` для
будущего Web Push (задача PWA-полировки, ADR будет дополнен при реализации бэкенда push).

`sw.ts` исключён из `tsconfig.app.json` (`exclude: ["src/sw.ts"]`), потому что типы
`ServiceWorkerGlobalScope`/`PushEvent`/`self` из lib `WebWorker` конфликтуют с `DOM` lib,
используемым остальным приложением; `esbuild` (внутри vite build) транспилирует файл без
строгой проверки типов — `tsc -b` в CI на нём не спотыкается, а `npm run build` всё равно
компилирует и бандлит воркер отдельным шагом (лог `PWA v1.3.0 ... Building src/sw.ts`).

`devOptions.enabled: false` — сервис-воркер не активируется в `vite dev` (только в
собранном `dist/`), поэтому офлайн-кэш и фоновая синхронизация проверяются исключительно
через `npm run build` + `vite preview`/деплой, не через обычный dev-сервер предпросмотра.

Иконки (`frontend/public/icons/*.png`) сгенерированы одноразовым Node-скриптом на базе
`sharp` из `frontend/public/icons/source.svg` (сплошной квадрат `#e87b2d` с буквой «S»);
`sharp` установлен как devDependency временно и удалён сразу после генерации — он не нужен
в рантайме и не должен постоянно висеть в `package.json` только ради разовой задачи. Если
понадобится перегенерировать иконки — тот же приём (см. этот ADR) с новым source.svg.

## 2026-07-06 — ADR-014: права групп — owner неявно всё, admin — по флагам; владелец не может выйти, пока в группе есть кто-то ещё

**Решение:** `has_permission()` (`backend/app/services/chat.py`) даёт `owner` любые права без
проверки флагов; `admin` — только те, что явно включены (`can_delete_messages`, `can_ban`,
`can_invite`, `can_pin`, `can_edit_info`); `member` не имеет админ-прав вообще, даже если
в БД у него выставлены флаги (они физически есть в `chat_members` под каждой ролью, но
читаются только для `admin`). Смену роли/флагов (`PATCH /chats/{id}/members/{user}`) может
делать только `owner` — иначе admin с `can_ban` мог бы сам себя повысить до owner. Владельца
нельзя кикнуть (`DELETE .../members/{owner}` → `403 cannot_remove_owner`), а сам owner не
может выйти через `DELETE .../members/{self}`, пока в группе остался хоть один участник
(`400 owner_must_transfer`) — передача владения отдельным действием пока не реализована
(нет UI/эндпоинта), поэтому явный запрет лучше молчаливого «группа осталась без владельца».

**Контекст:** без этого правила первый же admin с `can_ban=true` мог бы удалить owner'а или
назначить себя владельцем через `PATCH members`, либо owner мог случайно выйти из активной
группы, оставив её в противоречивом состоянии (`chats.owner_id` всё ещё указывает на
вышедшего пользователя, у которого `chat_members.left_at` не NULL).

**Последствия:** передача владения группой — отдельная будущая задача (нужна явная
`POST .../transfer-ownership` или подобный эндпоинт); до тех пор owner физически не может
покинуть группу, если в ней остался кто-то ещё, только удалить остальных участников до
последнего и тогда выйти (либо оставить группу как есть). Повторное приглашение вышедшего
участника (`POST .../members`) реактивирует его старую строку `chat_members`
(`add_or_reactivate_members`), не создаёт новую — из-за `UNIQUE(chat_id, user_id)`.

## 2026-07-05 — ADR-012: миграция 0001 читает замороженный снимок схемы, не живой db/schema.sql

**Решение:** `0001_initial_schema.py` больше не читает `db/schema.sql` напрямую — файл
скопирован «как было» в `backend/app/db/alembic/snapshots/0001_schema.sql`, и именно его
читает миграция. Дальнейшие изменения схемы (пример — `original_name` в `files`, добавленный
в этой же сессии) вносятся в `db/schema.sql` **и** оформляются новой миграцией (0002+, дельта
через `op.add_column`/`op.create_table` и т.п.), как и предполагала ADR-005.

**Контекст:** первая версия 0001 читала `db/schema.sql` в момент запуска миграции (`Path(...).read_text()`).
Это означало, что миграция не заморожена — любая последующая правка `db/schema.sql` меняла
поведение уже «выполненной» ревизии 0001 задним числом. Проявилось на практике: добавили
`original_name` в `db/schema.sql` и написали дельта-миграцию 0002 — CI упал с
`DuplicateColumnError: column "original_name" ... already exists`, потому что 0001 (читая
живой файл) уже создавала эту колонку сама, до того как до неё добирался 0002.

**Последствия:** `db/schema.sql` остаётся единственным источником истины для «схемы на
сейчас», но воспроизвести историю по одним миграциям можно только запустив их по очереди
от 0001 — сравнивать текущий `db/schema.sql` с содержимым `snapshots/0001_schema.sql`
напрямую бессмысленно (снимок — это схема на 2026-07-05, до колонки `original_name`).
Если потребуется «сжать» историю миграций в новую базовую — снова взять текущий
`db/schema.sql`, сделать новый снимок и новую ревизию 0001 (с явным сбросом истории).

Ещё один нюанс, всплывший здесь же: `ALTER TABLE ... ADD COLUMN` всегда добавляет колонку
физически последней, а `pg_dump --schema-only` печатает колонки в физическом порядке —
`schema-check` в CI сравнивает именно этот текст. Поэтому при добавлении колонки дельта-
миграцией она должна быть дописана в `db/schema.sql` **последней в таблице** (после
`created_at` и т.п.), а не туда, где логически «читается уместнее» — иначе тот же
`schema-check` падает на чисто косметическом несовпадении порядка колонок.

## 2026-07-05 — ADR-011: волновая форма голосовых — на клиенте, без декодирования на сервере

**Решение:** волновую форму голосового сообщения считает браузер во время записи
(`AnalyserNode.getByteTimeDomainData`, RMS каждый кадр, даунсемпл до 40 точек) и
отправляет вместе с файлом как JSON-массив (`files.waveform`, уже был в схеме).
Сервер (`app/api/files.py`) для `kind=voice` только валидирует и сохраняет присланный
массив — аудио (webm/opus) не декодирует и amplitude сам не считает.

**Контекст:** серверное декодирование аудио потребовало бы ffmpeg/аналог в контейнере
(на Render это лишняя системная зависимость и время сборки), тогда как в браузере
волновая форма — побочный продукт самой записи через Web Audio API, доступный без
дополнительных библиотек.

**Последствия:** волновая форма зависит от клиента, отправившего файл (доверенный ввод
пользователя, не критично для отображения). Если голосовое загружено не через штатный
recorder (гипотетически, через будущий API), `waveform` будет `null` — плеер использует
плейсхолдер (см. `VoicePlayer.tsx`).

## 2026-07-05 — ADR-010: два CI-гочи с asyncpg, найденные при первом реальном прогоне

**Решение:**
1. `0001_initial_schema` больше не гонит `db/schema.sql` одним `exec_driver_sql` —
   файл режется на отдельные операторы (учитывая `$$...$$` тела plpgsql-функций),
   потому что asyncpg отказывается готовить several команд в одном prepared statement
   (`cannot insert multiple commands into a prepared statement`).
2. `backend/tests/conftest.py` создаёт для тестов отдельный engine с `NullPool`,
   а не переиспользует `app.db.session.engine`: pytest-asyncio может запускать
   разные тесты в разных event loop'ах, а пул соединений SQLAlchemy держит
   asyncpg-соединение, привязанное к тому loop'у, где оно создано — переиспользование
   в другом loop'е падает с `got Future ... attached to a different loop`.
3. Лимитер slowapi (`app/core/limiter.py`) сбрасывается перед каждым тестом
   (`limiter.reset()`) — все тестовые запросы идут с одного IP, поэтому независимые
   тесты иначе исчерпывают общий rate-limit друг у друга.

**Контекст:** ни один из этих багов не проявляется при обычной ручной проверке
(`ruff`/`mypy`/юнит-тесты без БД) — все три всплыли только при первом прогоне
`schema-check`/`lint-test-backend` в CI против настоящего `postgres:16`, что и есть
основная причина держать эти джобы в CI, а не полагаться только на локальные проверки
(см. ADR-002).

**Последствия:** локально (без Postgres) эти тесты по-прежнему не запустить —
верификация только через CI. Если в будущем добавится ещё один raw-SQL блок с
несколькими командами — использовать тот же `_split_sql_statements`.

## 2026-07-05 — ADR-009: Фаза 2 реализует только личные (direct) чаты

**Решение:** `GET/POST /api/v1/chats`, весь код `app/services/chat.py` и фронтенд
(`ChatListPage`, `ChatPage`) работают только с `chat.type == 'direct'`. Группы
(`type='group'`), роли участников, инвайты — не реализуются в этой фазе.

**Контекст:** `docs/01-TZ.md` §4 явно относит группы к фазе 4; таблица `chat_members`
уже поддерживает роли/права (нужны для групп), но бизнес-логика на них пока не опирается.
UI-действия с сообщениями (ответ, реакция, правка, удаление) в `MessageRow.tsx` реализованы
как всплывающие иконки по наведению, а не полноценное контекстное меню — упрощение,
достаточное для 1-на-1 переписки; при добавлении групп (упоминания, системные сообщения)
потребуется полноценное меню действий.

**Последствия:** `build_chat_out`/`build_message_out` в `app/services/chat.py` при
`chat.type != 'direct'` возвращают вырожденные значения (`peer_*` пустые, заголовок —
`chat.title`) — заготовка под фазу 4, не более. Голосовые сообщения и кнопка микрофона
в композере — заглушка (`disabled`), реализация — фаза 3.

## 2026-07-05 — ADR-008: /sync упрощён до «новых сообщений с даты», без event-log

**Решение:** `GET /api/v1/sync?since=<ISO-время>` возвращает новые сообщения по всем
чатам пользователя, созданные после `since`, вместо генерального event-log с монотонным
`seq`, реплицируемым по всем типам событий (`reaction.updated`, `read.updated` и т.д.).
`seq` в WS-событиях (`app/ws/manager.py`) — счётчик в памяти процесса, уникальный в
рамках соединения, но не персистентный между рестартами.

**Контекст:** полноценный event-sourcing лог (таблица событий с монотонным seq на
пользователя, реплей всех типов событий после reconnect) — заметный объём работы,
не критичный для MVP с ≤100 одновременных WS-подключений: реакции/прочтения быстро
досчитываются обычным REST-поллингом чата при следующем открытии, а пропущенные
сообщения — самое важное — покрыты `/sync`.

**Последствия:** после длительного разрыва соединения статусы реакций/прочтений могут
на короткое время не совпадать до следующего запроса `GET .../messages`; если это
станет проблемой — реализовать персистентный event-log (таблица `events` + Redis
Stream) отдельной задачей.

## 2026-07-06 — ADR-013: демо-БД — Supabase, не Neon (отменяет ADR-007)

**Решение:** для тестового деплоя БД (и Storage) — Supabase, не Neon/Vercel Storage.
Заказчик явно попросил убрать Neon и использовать «только Vercel + Supabase». Vercel
по-прежнему раздаёт фронтенд из `frontend/`; API (FastAPI + WebSocket) всё так же не может
жить на самом Vercel (serverless-функции не держат постоянные WS-соединения, см.
`docs/02-ARCHITECTURE.md` §1) и разворачивается на Render, подключаясь к Supabase Postgres
через `DATABASE_URL` (`postgresql+asyncpg://`, host `db.<project>.supabase.co`, порт 5432
прямое подключение или 6543 через pooler в режиме transaction).

**Контекст:** ADR-007 (2026-07-05) переключал демо-БД на Neon по неверно понятой формулировке
«database-postgresql» — заказчик уточнил, что имел в виду обычный Supabase, уже
предусмотренный исходной архитектурой (`docs/02-ARCHITECTURE.md` §5) и `docs/03-SETUP-DEPLOY.md`
до правок этой сессии. Возврат к Supabase заодно устраняет неудобство двух аккаунтов
(Neon для БД + Supabase для Storage) — теперь один аккаунт закрывает и БД, и файлы.

**Последствия:** `docs/03-SETUP-DEPLOY.md` и `render.yaml` возвращены к Supabase-инструкции;
шаги Vercel Storage/Neon удалены. `STORAGE_BACKEND=supabase` в
`backend/app/services/storage.py` по-прежнему заглушка (`NotImplementedError`) — на
демо-деплое пока используется `STORAGE_BACKEND=local` (аватары/медиа не переживают
редеплой Render), реализация `SupabaseStorage` — отдельная задача.

## 2026-07-05 — ADR-007: демо-деплой — Vercel (фронт + Postgres через Neon), API отдельно (отменено ADR-013)

**Решение:** для тестового демо-деплоя БД поднимается через интеграцию Vercel Marketplace
(Neon Postgres, добавляется во вкладке Storage проекта на Vercel) — это и есть
«database-postgresql» из запроса заказчика. Фронтенд собирается и раздаётся Vercel
из `frontend/`. Backend (FastAPI + WebSocket) **не может жить на Vercel** (serverless-функции
не держат постоянные WS-соединения, см. `docs/02-ARCHITECTURE.md` §1) — API по-прежнему
разворачивается на Render/Railway (или временно — на VPS), подключаясь к той же Neon БД
через `DATABASE_URL` (драйвер `postgresql+asyncpg://`, для Neon — через pooler-хост).

**Контекст:** заказчик уточнил, что тестовый деплой должен использовать Vercel с базой
PostgreSQL. Нативный продукт «Vercel Postgres» упразднён в пользу маркетплейс-интеграций
(Neon/Supabase) — выбран Neon как ближайший аналог с бесплатным тиром и тем же диалектом
Postgres 16-совместимым протоколом.

**Последствия:** см. ADR-013 — заказчик уточнил, что предпочитает Supabase, запись отменена.

## 2026-07-05 — ADR-006: refresh-токен — схема «selector.secret» с ротацией

**Решение:** cookie `spi_refresh` хранит значение `"{session_id}.{secret}"`. `secret`
хэшируется (Argon2) и кладётся в `sessions.refresh_hash`; `session_id` — открытый
селектор для быстрого поиска строки без перебора. При каждом `/auth/refresh` секрет
перегенерируется и хэш перезаписывается (ротация), старое значение cookie становится
недействительным.

**Контекст:** Argon2-хэш — соленый и не допускает поиска «по значению» (обратное неверно
для probabilistic hashing), поэтому одного хэша на всю таблицу `sessions` недостаточно —
нужен открытый идентификатор строки (селектор), стандартный паттерн для refresh-токенов
(аналогично Django/Rails remember-me токенам).

**Последствия:** `sessions.refresh_hash` не является уникальным индексом для поиска;
поиск всегда идёт по `id`. Компрометация access-токена не даёт доступа к refresh —
они хранятся раздельно (JWT в памяти фронта, refresh — httpOnly cookie).

## 2026-07-05 — ADR-005: Alembic-миграция исполняет db/schema.sql как есть

**Решение:** единственная (пока) Alembic-миграция `0001_initial_schema` не дублирует DDL
через `op.create_table`, а читает и выполняет `db/schema.sql` целиком одним
`op.get_bind().exec_driver_sql(...)`. `schema-check` в CI поднимает две базы — одну через
`psql -f db/schema.sql`, другую через `alembic upgrade head` — и сравнивает `pg_dump
--schema-only` обеих; расхождение = красный CI.

**Контекст:** дублирование полной DDL-схемы (20+ таблиц, enum-типы, generated-колонки,
триггеры) в декларативном стиле Alembic рискует разойтись с эталонным `db/schema.sql`
(источник истины по требованию заказчика — «готовый SQL-скрипт создания БД»). Прогон
самого файла исключает расхождение конструктивно, а не проверкой постфактум.

**Последствия:** будущие изменения схемы вносятся в `db/schema.sql` и получают новую
Alembic-ревизию, которая применяет только дельту (новый `ALTER`/`CREATE` для добавленных
объектов), а не весь файл заново повторно.

## 2026-07-05 — ADR-004: ruff — отключены RUF001-003 и B008

**Решение:** в `backend/pyproject.toml` отключены `RUF001`/`RUF002`/`RUF003` (ambiguous
Cyrillic character) и `B008` (`Depends(...)` в аргументах по умолчанию).

**Контекст:** RUF001-003 массово ложно срабатывают на любом русском тексте (докстроки,
комментарии, строковые литералы) — несовместимо с ТЗ («интерфейс и код — на русском»).
B008 конфликтует с идиоматическим FastAPI-паттерном `param: T = Depends(...)`,
официально рекомендованным самим FastAPI.

**Последствия:** обычные опечатки/подстановки латиница↔кириллица эти правила больше не
ловят — полагаемся на ревью и рендеринг в браузере/IDE.

## 2026-07-02 — ADR-003: i18n без внешней библиотеки

**Решение:** словарь переводов RU/EN — собственный модуль `frontend/src/shared/i18n`
(типизированные ключи, хук `useT()`), без react-i18next.

**Контекст:** ТЗ требует RU как основной язык и EN вторым. Полноценный i18n-фреймворк
(плюрализация ICU, ленивые бандлы) избыточен для двух языков и увеличивает бандл;
типизированный словарь ловит отсутствующие ключи на этапе tsc.

**Последствия:** плюрализация решается функциями-хелперами (`plural(n, ...)` для русских
форм). Если языков станет больше трёх — мигрировать на react-i18next, структура ключей совместима.

## 2026-07-02 — ADR-002: локальная среда без Docker

**Решение:** разработка ведётся на машине без Docker Desktop; все проверки, требующие
PostgreSQL (alembic-миграции, интеграционные тесты, сверка с db/schema.sql), выполняются
в CI (GitHub Actions, сервис-контейнер postgres:16). Локально гоняются линтеры, tsc,
unit-тесты и сборка.

**Контекст:** на машине разработчика недоступны Docker и локальный PostgreSQL;
ставить полноценный сервер БД ради dev-цикла не требуется — CI покрывает
БД-зависимую часть на каждом push.

**Последствия:** compose-файлы (dev и prod) поддерживаются в репозитории и проверяются
на VPS/CI; unit-тесты backend пишутся так, чтобы не требовать живой БД (изоляция через
слои сервисов + тестовая SQLite несовместима с PG-фичами, поэтому для запросов с
tsvector/JSONB — только интеграционные тесты в CI).

## 2026-07-02 — ADR-001: структура и стек — строго по docs/02-ARCHITECTURE.md

**Решение:** монорепо `frontend/` (React 18 + Vite + TS + Zustand + TanStack Query) и
`backend/` (Python 3.12 + FastAPI + SQLAlchemy 2 async + Alembic), PostgreSQL 16 везде,
один WebSocket `/ws`, отправка сообщений через REST с `client_msg_id`.

**Контекст:** архитектура утверждена заказчиком, отклонения фиксируются здесь.

**Последствия:** любые изменения стека или протокола WS требуют новой записи ADR.
