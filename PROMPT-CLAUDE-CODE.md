Ты — ведущий fullstack-разработчик. Твоя задача — с нуля реализовать веб-мессенджер **SPI Messenger** до состояния готового к показу MVP, строго по документации проекта.

## Исходные материалы (прочитай ПЕРЕД началом работы)

1. `docs/01-TZ.md` — полное техническое задание: функционал, фазы, критерии приёмки. Это главный документ.
2. `docs/02-ARCHITECTURE.md` — утверждённая архитектура: стек, структура репозитория, протокол WS, профили деплоя. Не отклоняйся от неё без крайней необходимости; если отклоняешься — зафиксируй причину в `docs/DECISIONS.md`.
3. `db/schema.sql` — эталонная схема PostgreSQL 16. Модели SQLAlchemy должны ей соответствовать (проверяется в CI).
4. `design/mockups/` — макеты Figma (desktop и mobile). Интерфейс должен визуально соответствовать им: рассмотри изображения внимательно, извлеки точные цвета пипеткой, повтори компоновку, скругления, статусы «печатает…»/«записывает голосовое…», галочки доставки, флажки закладок, разделители дат.

## Ключевые требования (кратко)

- **Стек:** React 18 + Vite + TypeScript + Zustand + TanStack Query (frontend); Python 3.12 + FastAPI + SQLAlchemy 2 async + Alembic (backend). Монорепо по структуре из архитектуры.
- **БД:** PostgreSQL 16 везде — Supabase в демо, контейнер `postgres:16` на VPS. Драйвер asyncpg, миграции Alembic. Используй возможности Postgres: JSONB, tsvector с русской морфологией для поиска, `INSERT ... ON CONFLICT` для идемпотентности.
- **Mobile-first + PWA — высший приоритет.** Приложение будет использоваться в основном на iPhone: установка на домашний экран (manifest, apple-touch-icon, splash), standalone-режим, safe-area-insets, Web Push через VAPID (подписка только по явному жесту пользователя), offline-кэш оболочки и истории, очередь исходящих сообщений при отсутствии сети, Badging API. На десктопе — двухпанельный макет как на desktop-мокапе.
- **Темы:** тёмная (по макетам) и светлая (белый/оранжевый/тёмно-синий, стилистика spi-2015.ru) + системная. Все цвета — через CSS-переменные, тема переключается атрибутом на `<html>`.
- **Безопасность:** ни одного секрета в коде или git-истории. `.env` в `.gitignore`, `.env.example` со всеми ключами без значений. JWT access 15 мин + refresh в httpOnly cookie с ротацией. Rate limiting, CORS whitelist, санитизация Markdown, валидация загружаемых файлов.
- **Real-time:** один WebSocket `/ws`, события по протоколу из архитектуры; отправка сообщений через REST с `client_msg_id` (идемпотентность); reconnect + `/sync?since=`.

## Порядок работы — фазы

Работай строго по фазам из `docs/01-TZ.md` (раздел 4). Не начинай следующую фазу, пока текущая не удовлетворяет её критериям и не закоммичена.

- **Фаза 0 — Каркас.** Монорепо, структура из архитектуры, линтеры (ruff, mypy, eslint, prettier, tsc strict), pytest/vitest, GitHub Actions (lint-test + schema-check на PostgreSQL 16 в сервис-контейнере), `docker-compose.dev.yml` (postgres, redis, minio), `docker-compose.yml` (прод VPS), `.env.example`, дизайн-токены и ui-kit (кнопки, инпуты, аватары, пузыри сообщений — по макетам), скелет страниц с роутингом.
- **Фаза 1 — Auth и профиль.** Вход по e-mail-коду (в dev-режиме код выводится в лог, чтобы работать без SMTP), JWT + сессии с отзывом, страница профиля (аватар с кропом, имя, био, @username с проверкой уникальности), контакты по @username, настройки (тема, язык, приватность), блокировки.
- **Фаза 2 — Личные чаты.** Список чатов (закреп, архив, mute, счётчики непрочитанных, превью последнего сообщения — как на макете), диалог: real-time доставка, статусы ✓/✓✓, typing-индикатор, presence («был(а) в сети N минут назад»), reply с цитированием, реакции (двойной тап = ❤️), edit/delete (у себя/у всех), черновики с синхронизацией, пересылка, разделители дат, Markdown-форматирование.
- **Фаза 3 — Медиа.** Загрузка фото/видео/документов через StorageService (supabase|local), превью и лайтбокс, голосовые с волновой формой и скоростью 1x/1.5x/2x, vCard и геолокация, Saved Messages, закладки-флажки, глобальный поиск и поиск по чату (tsvector + GIN), медиа-архив чата (Медиа/Файлы/Ссылки/Голосовые), превью ссылок (OG).
- **Фаза 4 — Группы.** Создание, аватар/описание, добавление участников, инвайт-ссылки и QR, роли и гранулярные права админов, бан, закреплённые сообщения, упоминания @username, опросы, системные сообщения («X добавил Y»), альбомы.
- **Фаза 5 — PWA и полировка (MVP).** Push-уведомления end-to-end (включая iOS), offline-режим, Lighthouse PWA ≥ 90, обе темы отполированы по макетам, экспорт истории в JSON/HTML, страница «Активные сессии», кнопки звонков в UI с заглушкой «Скоро», экраны пустых состояний, seed-скрипт с демо-данными для показа (несколько пользователей и чатов, похожих на макеты).
- **Фаза 6 (только после явного подтверждения пользователя):** WebRTC-звонки, E2EE, стикеры/GIF, папки чатов.

## Git-процесс (обязателен)

- Инициализируй работу от текущего `main`. Пуш — в существующий origin (`git remote -v`; если remote не настроен — остановись и спроси у меня URL).
- **Коммить атомарно и часто**: каждая законченная подзадача = отдельный коммит. Conventional Commits (`feat:`, `fix:`, `chore:`, `docs:`, `test:`), сообщения на английском, тело — с пояснением «зачем».
- **Пушь после каждой завершённой фазы и минимум раз в 5–7 коммитов.** Перед push прогоняй линтеры и тесты; красный CI недопустим.
- Первый коммит фазы 0 — документация и каркас; тег `v0.1.0-mvp` после завершения фазы 5.
